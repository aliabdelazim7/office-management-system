# نظام إدارة المكتب المتكامل

منصة **ERP SaaS** متعددة المستأجرين لمكاتب الخدمات الحكومية والاستشارات المالية والقانونية — مصممة للسوق المصري والعربي.

> **حالة المشروع:** قيد التطوير النشط — Phase 1 (الأساس) مكتملة. راجع [خارطة الطريق](#خارطة-الطريق).

---

## نظرة عامة

كل مكتب يعمل كـ **Tenant مستقل** ببيانات معزولة تماماً: عملاء، موظفون، مستندات، حسابات. النظام يغطي دورة العمل الكاملة لمكتب الخدمات — من استقبال العميل، مروراً بالمستندات الحكومية وسير عمل الخدمات والمهام الميدانية، وصولاً إلى الفوترة والتحصيل والتنبيهات.

### الوحدات

| الوحدة | الوصف |
|---|---|
| **CRM** | ملف عميل 360° بالكيانات القانونية المصرية الستة |
| **خزنة المستندات** | تصنيف، تواريخ انتهاء، سجل نسخ، بنية جاهزة لـ OCR |
| **المستندات الحكومية** | السجل التجاري والفروع، البطاقة الضريبية، التأمينات، عضوية الاتحاد |
| **العقود** | قوالب بحقول ديناميكية، ملء تلقائي من بيانات العميل، توليد PDF |
| **محرك سير العمل** | الخدمات كمراحل بمسؤول وموعد وملفات — لا مجرد حقل حالة |
| **الموارد البشرية** | بيانات الموظفين، الوظائف، المرتبات |
| **الموظفون الميدانيون** | تجربة mobile-first مع تتبع GPS وصور إثبات وتقارير |
| **المالية** | فواتير، دفعات، مصروفات، ودفتر أستاذ لكل عميل |
| **محرك التنبيهات** | داخل النظام + بريد + واتساب |
| **المساعد الذكي** | استعلامات لغة طبيعية **مع التحقق من الصلاحيات** |

## المعمارية

```
apps/
  api/        NestJS — modular, DI, repository pattern, DTO validation
  web/        Next.js — App Router, RTL native, Arabic first
packages/
  database/   Prisma schema + كتالوج الصلاحيات المشترك + seed
  types/      عقود TypeScript مشتركة
```

### التقنيات

**الواجهة:** Next.js · React · TypeScript · TailwindCSS · React Query · Zustand · Zod
**الخادم:** NestJS · TypeScript · Passport JWT · class-validator
**البيانات:** PostgreSQL · Prisma
**التخزين:** S3-compatible (Supabase Storage / Cloudflare R2 / MinIO)

## الأمان

الأمان في هذا النظام مبني في الأساس، لا مضاف لاحقاً. القرارات الجوهرية:

### عزل المستأجرين لا يعتمد على انضباط كاتب الاستعلام

`PrismaService` يحمل امتداداً يحقن `tenantId` في **كل** استعلام قبل وصوله لقاعدة البيانات، ويقرأ السياق من `AsyncLocalStorage`. قائمة النماذج المشمولة **تُشتق من الـ DMMF وقت التشغيل** لا من قائمة يدوية — لأن انحراف قائمة يدوية عن السكيما يعني تسريباً صامتاً بين الشركات.

**إذا غاب سياق المستأجر، الاستعلام يفشل** ولا يُرجع صفوفاً.

### هوية المستأجر من التوكن حصراً

لا يوجد مسار `X-Tenant-ID` في النظام. قبول مستأجر يحدده العميل — خاصة إذا حُلّ قبل المصادقة — هو الطريقة القياسية لتسريب أنظمة multi-tenant. الحارس يتحقق أيضاً أن `user.tenantId` يطابق ادعاء التوكن.

### الرفض هو الافتراضي

أي مسار غير معلن كـ `@Public()` ولا يعلن صلاحياته المطلوبة → **يُرفض**. ديكوريتور منسي يصبح خطأ ظاهراً، لا نقطة نهاية مفتوحة صامتة.

### المصادقة

- كلمات المرور بـ bcrypt (cost 12)، ومقارنة وهمية عند عدم وجود المستخدم منعاً لتسريب التوقيت
- قفل الحساب بعد محاولات فاشلة متتالية
- **تدوير refresh tokens مع كشف إعادة الاستخدام**: استخدام توكن مُدوَّر يُبطل عائلة الجلسة كاملة
- التوكنات تُخزَّن كـ hash — تسريب قاعدة البيانات لا يعطي جلسات

### كتالوج صلاحيات واحد

٥١ صلاحية مُعرَّفة في مكان واحد (`@saas/database`)، تستهلكها الـ API والـ seed والواجهة. الواجهة ترسم مصفوفة الصلاحيات من الخادم بدلاً من قائمة مكررة تنحرف بصمت.

### الأدوار

| الدور | النطاق |
|---|---|
| `OWNER` | كل الصلاحيات — غير قابلة للتضييق (منعاً لقفل المكتب خارج إدارته) |
| `MANAGER` | إدارة التشغيل |
| `ACCOUNTANT` | المالية والتقارير |
| `EMPLOYEE` | المهام والعملاء المكلف بهم **فقط** |
| `VIEWER` | قراءة فقط |

## قاعدة البيانات

٣٧ نموذجاً، ١٩ enum، و٧٢ فهرساً على مسارات الاستعلام المتوقعة.

قرارات تصميمية مقصودة:

- **`tenantId` على كل كيان**، حتى الصفوف التي لا تُقرأ إلا عبر أب (`Payment` عبر `Invoice`). تكرار الحقل هو ثمن إمكانية إثبات العزل بشرط واحد وفرضه مركزياً.
- **`Decimal(14,2)` للمبالغ، لا `Float`** — الفاصلة العائمة الثنائية لا تمثل `0.10`، وهذا نظام يصدر فواتير.
- **ترقيم متسلسل من عدّاد لكل مكتب**، لا `COUNT(*)+1` الذي يتصادم تحت التزامن.
- **تفرّد داخل المكتب لا عالمياً** — نفس البريد قد يعمل في مكتبين.
- **`Payment` و `LedgerEntry` و `AuditLog` سجلات append-only** لا تُحذف.

## التشغيل محلياً

### المتطلبات

- Node.js 20+
- pnpm 9+
- قاعدة PostgreSQL (يُقترح [Supabase](https://supabase.com) أو [Neon](https://neon.tech) — الطبقة المجانية تكفي)

### الخطوات

```bash
pnpm install

cp .env.example .env
# املأ DATABASE_URL و DIRECT_URL ومفاتيح JWT

# ولّد مفاتيح JWT عشوائية:
node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"

pnpm db:generate
pnpm --filter @saas/database migrate
pnpm db:seed

pnpm dev:api   # http://localhost:4000/api/v1  ·  docs على /docs
pnpm dev:web   # http://localhost:3000
```

> **Supabase تحديداً:** يلزم متغيران — `DATABASE_URL` عبر المنفذ `6543` (transaction pooler) للتطبيق، و`DIRECT_URL` عبر `5432` للترحيل، لأن `prisma migrate` لا يعمل عبر pooler.

بيانات الدخول التطويرية تُطبع في نهاية `db:seed`. **موجودة في التطوير فقط** — السكربت يرفض العمل عند `NODE_ENV=production`.
